  <name>Task 1: Create registration server actions</name>
  <files>src/app/actions/registration.ts</files>
  <action>
    Create `src/app/actions/registration.ts` with "use server" directive. Follow the same patterns as `src/app/actions/reportes.ts` — create Supabase client, verify user auth, perform DB operations, revalidatePath, return ActionState.

    **Action 1: completeArrival(reporteId: string)**
    - Verify auth
    - `supabase.from("reportes").update({ llegada_completada: true }).eq("id", reporteId)`
    - `revalidatePath("/tecnico")`
    - Return `{ success: true }`

    **Action 2: completeSiteOverview(reporteId: string)**
    - Verify auth
    - `supabase.from("reportes").update({ sitio_completado: true }).eq("id", reporteId)`
    - `revalidatePath("/tecnico")`
    - Return `{ success: true }`

    **Action 3: checkFolioSitePhoto(folioId: string): Promise<{ exists: boolean; photoUrl?: string }>`**
    - Verify auth
    - Query: find any photo with etiqueta='sitio' across all reports for this folio:
      ```typescript
      const { data } = await supabase
        .from("reporte_fotos")
        .select("url, reportes!inner(folio_id)")
        .eq("reportes.folio_id", folioId)
        .eq("etiqueta", "sitio")
        .limit(1)
        .maybeSingle();
      ```
      Note: This uses the inner join syntax. If that doesn't work with the Supabase JS client, use a two-step approach:
      1. Get all reporte IDs for the folio: `supabase.from("reportes").select("id").eq("folio_id", folioId)`
      2. Query reporte_fotos with `.in("reporte_id", reporteIds).eq("etiqueta", "sitio").limit(1)`
    - Return `{ exists: !!data, photoUrl: data?.url }`

    **Action 4: saveEquipmentRegistration(equipoId: string, reporteEquipoId: string, data: RegistrationData)**
    - Verify auth
    - Validate data with Zod schema (import from validations/equipos.ts — see Task 2)
    - Build update object from non-null/non-empty fields only
    - Update equipos table: `supabase.from("equipos").update(updateData).eq("id", equipoId)`
    - Check if registration is complete:
      - Fetch the equipment row to check all required fields are filled (marca, modelo, numero_serie, capacidad, refrigerante, voltaje, fase, ubicacion — all non-null and non-empty)
      - Check photos: query `reporte_fotos` for this report's folio to see if equipo_general AND placa photos exist for this equipment
        - Get the folio_id from reporteEquipoId -> reporte_equipos -> reportes -> folio_id
        - Get all report IDs for that folio
        - Check for photos with equipo_id = equipoId AND etiqueta IN ('equipo_general', 'placa') across all those reports
      - If all fields filled AND both photos exist: `supabase.from("reporte_equipos").update({ registro_completado: true }).eq("id", reporteEquipoId)`
      - Otherwise: `supabase.from("reporte_equipos").update({ registro_completado: false }).eq("id", reporteEquipoId)`
    - `revalidatePath("/tecnico")`
    - Return `{ success: true }`

    **Action 5: checkRegistrationComplete(reporteId: string): Promise<boolean>**
    - Verify auth
    - Query: `supabase.from("reporte_equipos").select("registro_completado").eq("reporte_id", reporteId)`
    - Return `true` if ALL entries have `registro_completado === true` AND there's at least 1 entry
    - Return `false` if any entry has `registro_completado === false` or no entries exist

    **Action 6: updateRegistrationStatus(reporteEquipoId: string, equipoId: string, reporteId: string)**
    This is a helper called after a photo upload to re-evaluate registration completeness for a specific equipment. Same logic as the completeness check in Action 4:
    - Get equipment data to check fields
    - Check photos exist (equipo_general + placa)
    - Set registro_completado accordingly
    - `revalidatePath("/tecnico")`
    - Return `{ success: true, complete: boolean }`

    All actions should have proper error handling matching existing patterns (try/catch or error checking on Supabase responses).
  </action>
  <verify>
    Run `npx next build --webpack` to verify no TypeScript errors. Check that all 6 actions are exported and have correct signatures.
  </verify>
  <done>
    All 6 registration actions created and exported. Each follows the established auth -> validate -> mutate -> revalidate -> return pattern. Error handling matches existing codebase conventions.
  </done>
