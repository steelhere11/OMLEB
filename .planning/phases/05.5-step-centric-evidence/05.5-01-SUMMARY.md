# Phase 5.5 Plan 01: Step-Centric Evidence Redesign Summary

**One-liner:** Deduplicated stage-section evidence capture with step-organized PDF including inline photos, readings tables, and summary bar.

## What Was Done

### Task 3: Restructure PDF Data Pipeline (executed first)
- Added `reportePasoId` to `PhotoBase64` interface in pdf-utils.ts
- Updated `fetchAllPhotosAsBase64` to carry through `reporte_paso_id`
- Created `PdfStepData` interface with per-step `photosBase64` and `lecturas_meta`
- Added `orphanPhotosBase64` to equipment entries for photos without step links
- In report-pdf-button, photos are distributed into a `stepPhotosMap` keyed by step ID
- Extended Supabase query in page.tsx to include `lecturas_requeridas` from `plantillas_pasos`
- Passed `id` and `lecturas_meta` through the data pipeline

### Task 1: Create EvidenceStageSection and Redesign WorkflowStepCard
- Created shared `EvidenceStageSection` component at `src/components/shared/evidence-stage-section.tsx`
- Extracts unique stages from `evidencia_requerida`, sorts by canonical order (antes/durante/despues)
- Each stage section: color-coded header, guidance description bullets, single "Tomar foto" button, photo count badge, filtered PhotoGalleryRow
- Replaced duplicate per-item evidence buttons in WorkflowStepCard with single `<EvidenceStageSection>` component
- Removed unused `etapaColors`, `getPhotoCount`, and `PhotoGalleryRow` import

### Task 2: Apply Stage-Section Pattern to WorkflowCorrective
- Replaced per-item evidence buttons with shared `EvidenceStageSection` in corrective workflow
- Passed issue-specific photos and handlers through component props
- Removed unused `etapaColors`, `getPhotoCountForIssue`, and `PhotoGalleryRow` import

### Task 4: Redesign PDF Document Layout (Full Rewrite)
- Complete rewrite of `report-document.tsx` (738 lines added, 343 removed)
- **Page 1 layout:** Fixed header + info grid with highlighted "Problema Reportado" + summary bar
- **Summary bar:** Equipment count, preventivo/correctivo split, steps completed, photo count
- **Equipment cards:** Header with work type badge + progress counter
- **Step blocks:** Three rendering modes: full content (readings + notes + photos), completed single-line, incomplete single-line
- **ReadingsTable helper:** Parameter/Value/Range/Status columns with out-of-range warning indicators
- **StepPhotoGrid helper:** Groups photos by stage with color-coded labels (ANTES/DURANTE/DESPUES)
- **Orphan photos:** Rendered in "Fotos adicionales" section per equipment
- **Materials table:** Alternating row backgrounds
- **Page numbers:** All pages via `render` prop on footer Text element
- **Fixed footer:** Generated date (left) + page number (center) + company name (right)

### Task 5: Update Admin Report Detail for Step-Photo Consistency
- Grouped photos by `reporte_paso_id` within each equipment card
- `StepRow` now accepts optional `stagePhotos` prop
- Photos rendered inline with steps, grouped by stage with color-coded labels
- Orphan photos (no `reporte_paso_id`) shown in separate "Fotos adicionales" section
- Equipment entries without workflow steps retain original photo grid

## Deviations from Plan

None -- plan executed exactly as written.

## Decisions Made

| Decision | Context | Choice |
|----------|---------|--------|
| Stage deduplication | Multiple evidencia items with same etapa created duplicate buttons | Deduplicate by unique stage, combine descriptions as guidance bullets |
| Orphan photo fallback | Old photos have null reporte_paso_id | Render in fallback "Fotos adicionales" section per equipment |
| Fixed PDF header | Multi-page reports need consistent branding | Header fixed on all pages with company/client logos |
| Readings range display | lecturas_meta has rango_min/rango_max | Show structured table with parameter/value/range/status columns |
