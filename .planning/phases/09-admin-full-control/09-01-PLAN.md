# Plan 09-01: Database Migration + Types + Cascade Delete Actions

<objective>
Create the database migration for photo review status, admin notes, and comments table. Update TypeScript types. Implement all server actions for cascade deletes, photo management, step editing, and equipment info editing.
</objective>

## Context

This is the foundation plan for Phase 9: Admin Full Control. All other plans (09-02 through 09-06) depend on the server actions and types created here.

### Current State
- Phase 8 migration added: capacidad, refrigerante, voltaje, fase, ubicacion on equipos; llegada_completada, sitio_completado on reportes; registro_completado on reporte_equipos
- reporte_fotos already has: tipo_media ('foto'|'video'), etiqueta CHECK, reporte_paso_id
- FK constraints use RESTRICT (sucursales ← folios, equipos ← reporte_equipos, equipos ← folio_equipos)
- reportes → reporte_equipos, reporte_fotos, reporte_materiales, reporte_pasos all CASCADE on delete

### Architecture Decision: Soft Cascade via Server Actions
We do NOT change FK constraints to CASCADE on the parent tables. That's dangerous — accidental sucursal delete would wipe all history. Instead, keep RESTRICT and implement explicit admin cascade actions that delete children first in a controlled sequence.

### Task 1: Create migration SQL file
**File:** `supabase/migration-09-admin-control.sql` (NEW)

Create the SQL migration with:

1. **Photo review status column on reporte_fotos:**
```sql
ALTER TABLE public.reporte_fotos
  ADD COLUMN IF NOT EXISTS estatus_revision text NOT NULL DEFAULT 'pendiente'
  CHECK (estatus_revision IN ('pendiente', 'aceptada', 'rechazada', 'retomar'));
```

2. **Admin notes on photos:**
```sql
ALTER TABLE public.reporte_fotos
  ADD COLUMN IF NOT EXISTS nota_admin text;
```

3. **Admin comments table:**
```sql
CREATE TABLE IF NOT EXISTS public.reporte_comentarios (
  id          uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  reporte_id  uuid NOT NULL REFERENCES public.reportes (id) ON DELETE CASCADE,
  equipo_id   uuid REFERENCES public.equipos (id) ON DELETE SET NULL,
  autor_id    uuid NOT NULL REFERENCES public.users (id) ON DELETE RESTRICT,
  contenido   text NOT NULL,
  created_at  timestamptz NOT NULL DEFAULT now()
);
CREATE INDEX IF NOT EXISTS idx_reporte_comentarios_reporte ON public.reporte_comentarios (reporte_id);
```

4. **RLS for reporte_comentarios:**
- Admins: full access (FOR ALL)
- Technicians: read-only (FOR SELECT) for feedback visibility

5. **Add COMMENT ON statements** for documentation.

### Task 3: Create cascade delete server actions
**File:** `src/app/actions/admin-delete.ts` (NEW)

All actions must:
- Verify caller is admin role via getUser() check
- Use service role client for deletions that need to bypass RLS
- Fetch photo URLs before deleting rows (for storage cleanup)
- Delete storage files using supabase.storage.from('fotos').remove([paths])
- Revalidate affected paths after delete

**Actions:**

1. `adminDeleteReport(reporteId: string)`:
   - Fetch all reporte_fotos URLs for this report
   - Delete storage files for each photo/video
   - Delete the reporte row (CASCADE auto-deletes: reporte_equipos, reporte_fotos, reporte_materiales, reporte_pasos, reporte_comentarios)
   - Revalidate: /admin/reportes, /admin/folios

2. `adminDeleteFolio(folioId: string)`:
   - Fetch all reportes for this folio
   - For each reporte, fetch reporte_fotos URLs and delete storage files
   - Delete all reportes (CASCADE cleans children)
   - Delete the folio itself (CASCADE auto-deletes: folio_asignados, folio_equipos)
   - Revalidate: /admin/folios, /admin/reportes

3. `adminDeleteEquipo(equipoId: string)`:
   - Delete reporte_equipos rows where equipo_id matches (detach from reports)
   - Delete folio_equipos rows where equipo_id matches
   - reporte_fotos: equipo_id is ON DELETE SET NULL, so it auto-nullifies
   - Delete the equipo itself
   - Revalidate: /admin/equipos, /admin/reportes

4. `adminDeleteSucursal(sucursalId: string)`:
   - Fetch all folios for this sucursal
   - For each folio, run the delete folio logic (storage cleanup + delete reports + delete folio)
   - Fetch any orphan reports directly referencing sucursal_id
   - For each orphan report, run delete report logic
   - Delete the sucursal itself (CASCADE auto-deletes all equipos)
   - Revalidate: /admin/sucursales, /admin/folios, /admin/reportes, /admin/equipos

Use the existing Supabase client patterns from `src/lib/supabase.ts` (createClient for server, createServiceClient for admin operations). Look at existing actions in `src/app/actions/` for patterns.
