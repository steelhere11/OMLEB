    1. Create `src/lib/validations/equipos.ts` using the Zod 4 schema from RESEARCH.md:
       - sucursal_id: required UUID string
       - numero_etiqueta: required string, min 1, max 100
       - marca, modelo, numero_serie, tipo_equipo: optional strings, max 100 (use `.optional().or(z.literal(""))` for empty string handling since form fields send empty strings)

    2. Create `src/app/actions/equipos.ts` with three server actions:

       **createEquipo(_prevState: ActionState | null, formData: FormData): Promise&lt;ActionState&gt;**
       - Admin role check
       - Extract all fields from formData (sucursal_id from hidden input, not from URL)
       - Validate with equipoSchema.safeParse()
       - Insert into equipos: all validated fields + `revisado: true` (admin-created equipment is already reviewed)
       - Do NOT set `agregado_por` for admin-created equipment (leave null — it tracks tech-added equipment only)
       - revalidatePath(`/admin/equipos/${result.data.sucursal_id}`) + redirect

       **updateEquipo(id: string, _prevState: ActionState | null, formData: FormData): Promise&lt;ActionState&gt;**
       - Admin role check, validate (exclude sucursal_id from update — equipment stays in same branch)
       - Update by id. If editing tech-added equipment, set `revisado: true` (admin has reviewed it)
       - revalidatePath + redirect back to branch equipment page

       **deleteEquipo(_prevState: ActionState | null, formData: FormData): Promise&lt;ActionState&gt;**
       - Admin role check
       - Delete by id from formData
       - Handle FK constraint error (code "23503"): "No se puede eliminar este equipo porque tiene reportes asociados."
       - On success: revalidatePath and return success state

       - "Equipos" page heading

       - If branch not found, call notFound()
       - Fetch equipment for this branch: `supabase.from("equipos").select("*").eq("sucursal_id", sucursalId).order("numero_etiqueta")`
       - Show branch name in heading: "Equipos — {branch.nombre} ({branch.numero})"
       - Back link to /admin/equipos

    Folios are the most complex entity: they reference branches, clients, and users (cuadrilla assignment). The folio number is auto-generated by the database trigger.

    1. Create `src/lib/validations/folios.ts` using the Zod 4 schema from RESEARCH.md:
       - sucursal_id: required UUID string
       - cliente_id: required UUID string
       - descripcion_problema: required string, min 1, max 2000
       - All Spanish error messages

    2. Create `src/app/actions/folios.ts` with two server actions:

       **createFolio(_prevState: ActionState | null, formData: FormData): Promise&lt;ActionState&gt;**
       - Admin role check
       - Validate with folioSchema.safeParse()
       - Get selected user IDs: `formData.getAll("usuarios") as string[]`
       - If no users selected: return `{ error: "Debe asignar al menos un tecnico al folio" }`
       - Insert folio (numero_folio auto-generated by DB trigger): `supabase.from("folios").insert({ sucursal_id, cliente_id, descripcion_problema }).select("id").single()`
       - Insert assignments: `supabase.from("folio_asignados").insert(userIds.map(uid => ({ folio_id: folio.id, usuario_id: uid })))`
       - Handle folio insert error and assignment error separately (folio created but assignment failed = partial success message)
       - revalidatePath("/admin/folios") + redirect("/admin/folios")

       **updateFolio(id: string, _prevState: ActionState | null, formData: FormData): Promise&lt;ActionState&gt;**
       - Admin role check, validate
       - Update folio table (sucursal_id, cliente_id, descripcion_problema)
       - For user reassignment: delete all existing folio_asignados for this folio, then insert new assignments
       - `supabase.from("folio_asignados").delete().eq("folio_id", id)` then insert new ones
       - revalidatePath + redirect
       - Do NOT update estatus in Phase 2 (view-only, status changes come from Phase 3)

    3. Create `src/app/admin/folios/page.tsx` (Server Component):
       - Fetch folios with related data: `supabase.from("folios").select("*, sucursales(nombre, numero), clientes(nombre), folio_asignados(usuario_id, users(nombre, rol))").order("created_at", { ascending: false })`
       - Table columns: Numero Folio, Sucursal (nombre + numero), Cliente (nombre), Estatus (color-coded badge), Equipo asignado (count of assigned users), Fecha, Acciones (Editar link)
       - Status badges:
         - abierto: blue badge
         - en_progreso: yellow badge
         - completado: green badge
         - en_espera: orange badge
       - Spanish labels for status: "Abierto", "En Progreso", "Completado", "En Espera"
       - "Crear Folio" button linking to /admin/folios/nuevo
       - Empty state

         - Cliente dropdown (Select component): shows client nombre
         - Descripcion del problema (Textarea component, required)
         - User assignment section: "Asignar Equipo (Cuadrilla)" heading, then a list of checkboxes for each user (tecnico/ayudante only — filter out admins)
         - Each checkbox shows user nombre and rol badge
         - At least one user must be checked (client-side hint, server validates too)
         - Form fields: sucursal_id, cliente_id, descripcion_problema, usuarios[] (checkbox name="usuarios" value={user.id})

    5. Create `src/app/admin/folios/[id]/editar/page.tsx` (Server Component):
       - Fetch folio by id with its current assignments: `supabase.from("folios").select("*, folio_asignados(usuario_id)").eq("id", id).single()`
       - Fetch branches, clients, users (same as create page)
       - notFound() if folio not found
       - Render EditFolioForm with folio data, branches, clients, users, and current assignment IDs

    6. Create `src/app/admin/folios/[id]/editar/edit-form.tsx` ("use client"):
       - Accept props: folio, branches, clientes, users, currentAssignmentIds (string[])

       - Pre-populated dropdowns (defaultValue for selects)
       - Pre-checked checkboxes for currently assigned users
       - Show folio number as read-only info (auto-generated, not editable)
       - Show current estatus as read-only badge (not editable in Phase 2)
